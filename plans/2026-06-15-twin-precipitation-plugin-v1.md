# Twin Precipitation Plugin (Rain + Snow)

## Objective

Add live-data-driven precipitation to the offshore-wind digital twin as a self-contained, detachable plugin module rather than inline logic in the already-monolithic story file `storybook-webgpu/src/ocean/GlobeWaterproOcean-Story.tsx`. Rain is the primary deliverable; snow is a bonus served by the same system. The effect must be GPU-cheap, read correctly from orbit down to first-person, follow the camera so scene size is irrelevant, and react to the MET precipitation/wind/temperature already carried in each `MetSample` with no synthesized values (honouring the project no-fallback / no-mock rule). The success shape: the feature mounts and unmounts via a single JSX line in the story's Content component, owns all its own uniforms/buffers/compute/cleanup, and survives the planned story refactor untouched because nothing scene-specific leaks into it.

## Implementation Plan

- [ ] 1. Create a new detachable module folder `storybook-webgpu/src/weather/` (outside `packages/` to keep the ocean PR-able per the standing upstream constraint, and separate from `ocean/` so the future refactor can move the whole effect by moving one directory). Establish the plugin contract as a pure factory returning a `WeatherSystem` value (its `Object3D` to add to the scene, an `update` method taking camera + delta, a `sync` method taking plain CPU values, and a `dispose`), deliberately mirroring the `createCloudField` factory shape in `storybook-webgpu/src/ocean/cloud-coverage.ts:115` and its `sync`/`dispose` at `cloud-coverage.ts:262`. Add the mandatory five-line purpose header to every new file per the repo convention.
- [ ] 2. Build the GPU compute particle core in `storybook-webgpu/src/weather/createPrecipitationSystem.ts`: allocate persistent `instancedArray` storage buffers for per-drop position and velocity plus a ripple-time buffer for splashes, then a compute node built with the TSL `Fn`-then-`compute` pattern that integrates velocity, recycles drops that exit the volume by wrapping them via modulo, and an instanced streak material reading the position buffer in its `positionNode`. Use the official three.js example `webgpu_compute_particles_rain` as the reference implementation; target a configurable cap around 25k–50k drops drawn as billboarded streak quads with depth-write disabled and additive-style transparency.
- [ ] 3. Anchor the drop volume to the camera in a local frame so ECEF magnitudes (~6.37e6 m) never enter the buffers: place the system's group at the camera ECEF position each `update`, and re-derive the local East-North-Up basis from the camera position every frame the same way the FPS rig does (see the per-frame ENU note at `storybook-webgpu/src/ocean/GlobeWaterproOcean-Story.tsx:725`). Gravity pulls drops along geocentric down (negated normalized camera position), so rain falls toward the planet, not toward world minus-Y.
- [ ] 4. Map MET fields to system uniforms inside the factory's `sync`: precipitation in mm/h drives visible-drop count and opacity (zero or null precipitation disables the system entirely, never a token shower); `windSpeed` plus `windFromDirection` resolve to a horizontal ENU wind vector added to drop velocity and used to tilt the streak quads, reusing the same wind convention the turbine yaw already consumes at `GlobeWaterproOcean-Story.tsx:3676`; `airTemperature` (and `symbolCode` sleet/snow variants) select rain-versus-snow mode. All three values come straight from `MetSample` as surfaced in `examples/ocean-globe-waterpro-demo/ui/useMetForecast.ts:60`.
- [ ] 5. Write the thin R3F wrapper component `storybook-webgpu/src/weather/Precipitation.tsx` that owns mounting only, modelled on `storybook-webgpu/src/ocean/CloudLayer.tsx:26`: it builds the system once with `useMemo`, dispatches the compute node through the WebGPU renderer's `computeAsync` inside a default-priority `useFrame` (which runs before the existing post-processing render registered at priority 1, `GlobeWaterproOcean-Story.tsx:3584`), calls `update` with the live camera each frame, and calls the factory `sync` from its props. The renderer is read via the R3F `gl` state, which `WebGPUCanvas` already provides as a real WebGPU renderer through its async `gl` initializer in `storybook-webgpu/src/components/WebGPUCanvas.tsx`.
- [ ] 6. Gate visibility and density by camera altitude so drops are not paid for when they would be sub-pixel: reuse the existing overview/altitude signal (`useOverviewLOD` consumed at `GlobeWaterproOcean-Story.tsx:2177`) and fade drop opacity to zero across an altitude band, consistent with how the scene already fades the ocean patch and inscatter with altitude. At orbit the particle layer renders nothing; near the surface it is full strength.
- [ ] 7. Integrate into the story with the smallest possible footprint: add exactly one conditional mount line for the `Precipitation` wrapper next to the existing cloud mount at `GlobeWaterproOcean-Story.tsx:3596`, gated on effective precipitation greater than zero, and extend the Content props interface (`GlobeWaterproOcean-Story.tsx:2023`) with a nullable `precip` and a nullable `airTemperature`, reusing the already-present `windHeading` and `windSpeed` props. No effect logic lives in the story — only the mount and the prop pass-through.
- [ ] 8. Wire the MET sample into the new props from the deploy host at `examples/ocean-globe-waterpro-demo/main.tsx:663`, alongside the existing `windSpeed` and `clockMs` props, passing `sample` precipitation and air temperature from the same `sampleAt(selected)` result already computed at `main.tsx:437`, so precipitation tracks the forecast scrubber on the identical interpolated hour as sun position and wave state.
- [ ] 9. Add splash ripples for the offshore case using the analytic sea surface rather than a depth pre-pass: when a drop crosses local sea level, spawn a short-lived ripple via the ripple-time buffer and a small ring instance, which is cheaper than the reference example's render-target collision. Explicitly defer over-land/photogrammetry splash (FPS-on-deck/terrain height sampling) as a later phase, documenting the limitation rather than faking ground contact.
- [ ] 10. Add an optional, backend-agnostic scene-response layer owned by the same plugin: expose uniforms that, scaled by precipitation, slightly darken and desaturate the sky/atmosphere, raise ocean wetness/roughness, and tighten visibility/fog, since perceived rain intensity comes more from scene response than from counted drops. Keep this opt-in and wired through the plugin's own uniforms so it can ship after the particle core and so it can run even when the particle path cannot.
- [ ] 11. Parameterize snow as a mode of the same system, not a fork: reuse the identical buffers and compute, switching fall speed (slow), adding lateral sway via curl or sine drift plus tumble, swapping the streak quad for a soft round sprite, and disabling splash. Select the mode from `airTemperature`/`symbolCode` so a real sub-zero forecast yields snow automatically; expose a manual override for Storybook where no MET feed exists.
- [ ] 12. Add a WebGPU-backend guard, controls, and documentation: because the particle path needs the WebGPU compute backend (unavailable under the WebGL2 fallback that `WebGPUCanvas` falls back to, and absent on the WebGL renderer), detect the backend and render the particle layer only when compute is available, allowing the scene-response layer to remain. Add Storybook leva controls for manual intensity/wind/mode when no MET feed is present, and write a short module README in `storybook-webgpu/src/weather/` describing the contract and the deferred items, in the spirit of the existing ocean PORT-STATUS note.

## Verification Criteria

- Removing the single mount line at the cloud-mount site leaves the build green and the scene rain-free, proving the effect is fully detachable and adds no required logic to the story.
- With a live or mocked-in-Storybook precipitation above zero, drops are visible near the surface and the visible count and opacity scale monotonically with the precipitation value; precipitation of zero or null produces zero draws, not a faint shower.
- Scrubbing the forecast timeline changes rain intensity in lockstep with the existing sun and wave changes, confirming it reads the same `sampleAt` hour.
- Drops fall toward the planet (geocentric down) and lean with wind direction matching the turbine yaw indicator, at all camera headings and at both the equator and high latitude (camera-local ENU correctness, no ECEF jitter in the drops).
- Pulling the camera to orbit altitude fades the particle layer to nothing with no popping, and returning to surface restores it.
- A sub-zero `airTemperature` (or a snow `symbolCode`) switches the same system to snow behaviour with no separate component mounted.
- On a WebGL2-fallback session the particle layer is cleanly absent (no errors, no fake particles) while the optional scene-response layer, if enabled, still reflects precipitation.
- Frame time with the particle layer active at full intensity stays within a small delta of the no-rain baseline on the WebGPU backend (single compute dispatch plus one instanced draw).

## Potential Risks and Mitigations

1. **Compute backend unavailable under the WebGL2 fallback or plain WebGL renderer.** The particle core depends on WebGPU compute, which `WebGPUCanvas` does not provide when it falls back to WebGL2.
   Mitigation: detect the backend at mount and render the particle layer only when compute exists; keep the data-driven scene-response layer backend-agnostic so weather still reads on fallback. Do not silently substitute a fake CPU shower (no-fallback rule).

2. **ECEF float32 precision destroying the drop volume.** Storing drop positions at ~6.37e6 m magnitude collapses small inter-drop deltas and jitters.
   Mitigation: keep all buffer coordinates in a small camera-local frame, translate the group to the camera each frame, and re-derive the ENU basis per frame exactly as the FPS rig does.

3. **Transparent drops compositing incorrectly through the aerial-perspective post pass.** The scene is rendered through a post pipeline at priority 1, so naive transparency or depth-write can occlude or be occluded wrongly.
   Mitigation: disable depth-write on drops, draw after opaque geometry, dispatch compute at default priority before the post render, and verify ordering against the ocean and atmosphere as an explicit test.

4. **Wind-direction convention mismatch.** MET `windFromDirection` is the bearing the wind blows from; using it directly would tilt drops backwards.
   Mitigation: convert to a blowing-to vector and reuse the same convention the turbine yaw already applies, validating drop lean against the yaw indicator.

5. **Effect re-coupling to the monolith during the future refactor.** Convenience imports from the story could quietly tie the plugin back to scene internals.
   Mitigation: forbid story-internal imports in the module; the only inputs are the camera, plain CPU values, and the renderer, all passed in — enforced by keeping the module in its own folder with its own README contract.

6. **Particle cost creeping with intensity.** Naively spawning more drops for heavy rain could regress frame time.
   Mitigation: keep a fixed buffer cap and modulate only the visible fraction and opacity by precipitation, so cost is constant regardless of intensity; lean on scene-response for the heaviest-rain reading.

## Alternative Approaches

1. Inline particle code directly in the story Content. Rejected: it deepens the monolith the user explicitly wants to avoid and would have to be extracted again during the refactor.
2. CPU instanced-mesh drops with per-frame matrix updates (the Codrops 2025 R3F approach). Simpler and backend-agnostic so it would also run under WebGL2, but CPU-bound and capped at a few thousand drops; viable as the scene-response companion or a fallback-session visual, not as the primary engine.
3. Pure screen-space full-screen rain/lens-droplet shader with no 3D particles. Cheapest and infinitely scalable, but less convincing at first-person installation scale; best kept as a future POV-only enhancement layered on top, not the core.
4. Fold precipitation into the existing cloud raymarch in `packages/clouds`. Rejected: heavyweight, couples the twin effect into a shared upstream package, and violates the constraint that this work stay inside `storybook-webgpu/` and `examples/`.
